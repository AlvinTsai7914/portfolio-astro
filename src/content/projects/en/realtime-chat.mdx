## Project Overview

A practice project. Built a lightweight multi-room chat system with realtime messages, read receipts, typing indicators, and file sharing. Focus was on WebSocket connection management and message reliability.

## My Role

Solo full-stack:

- Frontend React + Zustand for realtime state
- Backend Express + Socket.io for multi-room broadcast
- Redis Pub/Sub for cross-node message distribution
- S3 for file upload/download

## Technical Highlights

### Reconnection and Message Reliability

WebSocket disconnects are normal. The client records the last received message ID and sends `lastSeenId` to the server on reconnect; the server backfills missing messages. Messages sent while offline are cached in IndexedDB and resent once the connection recovers.

### Read Receipts

Don't send a read receipt per message (traffic explodes). Use a "throttled last-read pointer" — once a user lingers on a message for 2 seconds, send one update. Render with two-line indicators (your read / their read) for visual clarity.

## Lessons Learned

First serious realtime system. Deepest takeaway: there's no silver bullet for distributed consistency — read receipts, message ordering, cross-device sync each force tradeoffs between performance and accuracy. Redis Pub/Sub is simple but loses messages; production would need Kafka or similar persistent queues.
